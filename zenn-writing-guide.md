# Zenn記事執筆ガイドライン🐈

このドキュメントは、R-524（@r5ni4）のZenn記事執筆時のトーン・スタイル・ルールをまとめたものである。
記事を書きながら随時更新していくつもりであり、完璧に踏襲するつもりもないので緩めにやっていく


## 基本方針

### ターゲット読者

- **初学者、若手、新人エンジニア**
- Zennを使い始めたばかりの人
- ツールのセットアップ方法を知りたい人
- 新しい技術に興味がある学習者

### 記事の目的

- 自分の学びを記録する
- 同じ課題に直面している人を助ける
- 完璧じゃなくても、等身大の学びを共有する
- どんなに小さくて些末な内容でもOK
- アウトプットの練習
- 他の人と違った角度、差別化したスタンス

## トーン・文体

### 基本トーン

- **技術記事らしからぬ気軽なトーン**
- 親しみを持てるように、筆者の主観を添える
- 漢字とひらがなのバランスに気を遣い、ややひらがな多めに
- やりすぎないように気をつけつつも文章の端々にユーモアを仕込む
- ちょっと笑えるポイントは極力作る
- 堅苦しくない、友達に話すような口調
- でも、最低限の礼儀は守り誰も不快にさせないことを徹底

### 語尾のバリエーション

```
✅ 使う：
- 〜だよ
- 〜だね
- 〜かな
- 〜だぜ
- 〜しよう
- 〜してみた

❌ 使わない：
- 〜である（堅い）
- 〜ございます（丁寧すぎ）
- 〜だろうか（仰々しい）
```

### 感情表現

- **正直な気持ちを書く**
  - 「困った」
  - 「うれしい」
  - 「意外だった」
- **失敗も包み隠さず**
  - 「やらかした」
  - 「詰んだ」
  - 「ミスった」

### 例文

```markdown
# ✅ 良い例
スラッグを指定せずに投稿したら、ランダムな文字列になってもた
でも記事の質の方が重要なので気にしない気にしない

# ❌ 悪い例
スラッグを指定しなかったため、ランダムな文字列が割り当てられました。
しかし、問題はありません。記事の品質の方が重要です。
```

## 構成ルール

### 必須セクション

1. **きっかけ**
   - なぜこの記事を書こうと思ったか
   - 実体験ベースで
   - 読者が共感できる導入

2. **こんな人向け**
   - 箇条書きで3-5個
   - 読者が「自分のことだ」と思える内容

3. **本文**
   - 見出しで整理
   - スクリーンショット多め
   - コードブロックは適宜

4. **まとめ**
   - 要点を3-5個に絞る
   - 絵文字で読みやすく

### セクションの書き方

#### きっかけ

```markdown
## きっかけ

先日、〇〇をやってみた。

〇〇だと思ってたけど、実は△△だった。
この辺り理解が浅いのでついでに調べてみることに。

調べてみたら、こういうことらしい。
というわけでやってみた。
```

#### こんな人向け

```markdown
## こんな人向け（だよ）

- 〇〇を使い始めたばかり
- △△が分からない
- □□したい
```

タイトルは「こんな人向け」「こんな人向けだよ」「読んでほしい人」など、バリエーションOK。

#### まとめ

```markdown
## まとめ

- ポイント1 
- ポイント2 
- ポイント3 

〇〇すると△△できる。
□□してみよう！
```

## ねこ要素 🐈

### ねこの落書きの使い方

記事の要所要所に**手書きのねこイラスト**を挿入する。

#### 挿入タイミング
- **感情の変化があるとき**
  - 困った → 困り顔のねこ
  - 解決した → 嬉しそうなねこ
  - 驚いた → 目を丸くしたねこ

- **重要なポイント**
  - 注意喚起 → 真面目な顔のねこ
  - ここが肝心 → ドヤ顔のねこ

- **息抜き**
  - 長い説明の後 → リラックスしてるねこ
  - 難しい概念の後 → 癒し系ねこ

#### ねこのセリフ

```markdown
![困り顔のねこ](画像パス)
*どうすればいいんだ...🐈*

![嬉しそうなねこ](画像パス)
*やったぜ！*

![ドヤ顔のねこ](画像パス)
*Git連携、最高だな...*
```

### ねこの描き方の特徴

- シンプルな線画
- 手書き感
- 表情豊か
- 状況に合わせたポーズ

## スクリーンショット

### 撮り方

- **UI全体よりも該当箇所を拡大**
- 不要な部分はトリミング
- 矢印や枠で強調してもOK

### キャプション

```markdown
![Cursorのチャット画面](画像パス)

# 説明が必要なら追加
モード選択の隣に、なんかあるぞ？
```

### ファイル名

```bash

# ✅ 良い例

```

### コメントの使い方

**初心者向け記事では、コードに説明を入れると親切**

コードブロック内に説明コメントを入れることで、読者が「このコマンド何してるの？」と迷わない。

```bash
# スラッグを指定して記事作成
npx zenn new:article --slug article-name

# リポジトリをクローン
git clone https://github.com/user/repo.git
```

コメントは日本語でOK。ただし、自明なコードには無理にコメント不要。

```bash
# ❌ 冗長（なくても分かる）
git add .  # ファイルを追加
git commit -m "記事を追加"  # コミット
git push  # プッシュ

# ✅ コメントなしでもOK
git add .
git commit -m "記事を追加"
git push
```

**コメントを入れた方がいい場面**：
- 複雑なコマンドの説明
- 初心者が躓きそうなポイント
- オプションの意味を補足したい時

**コメント不要な場面**：
- 自明なコマンド
- 本文で既に説明している内容

## 絵文字の使い方

### 使う場所

- 見出し（任意）
- 箇条書きのポイント
- 感情表現
- 注意喚起

### よく使う絵文字

```
感情：🤔🙃🫠🫥🤕✌️
```

### 使いすぎ注意

## 失敗談の書き方

### 基本方針

- **失敗を隠さない**
- むしろ積極的に書く
- 読者に「自分も同じミスしそう」と思ってもらう

### テンプレート

```markdown
## 私の失敗談

〇〇したら、△△になっちゃった😭

[ねこの落書き：ショック受けてる]

でも大丈夫！
- 〇〇すれば解決できる
- 実は△△だった
- 次からは□□しよう

### 教訓
**〇〇は最初が肝心**
でも、やらかしても死なない！
```

### 失敗の例
- スラッグがランダムになった
- 設定ミスで動かなかった
- 遠回りした
- 最初は理解できなかった

## フロントマター

### テンプレート

```yaml
---
title: "【初心者向け】〇〇の方法"
emoji: "🐈"  # 記事の内容に合わせて
type: "tech"  # tech or idea
topics: ["cursor", "ai", "初心者"]  # 最大5つ、小文字推奨
published: false  # 公開時は true
published_at: 2025-11-26 09:00  # 自動公開の日時（任意）
---
```

### emoji の選び方

- 記事の内容を象徴するもの
- ねこ、どうぶつ関連や食べ物など可愛い系もOK：🐈🌽🥕🥪
- 分かりやすいアイコン：💻📝🔧

### topics のルール

- 小文字で統一
- 具体的なツール名・技術名
- 「初心者」「初期設定」などのタグも有効
- 最大5つまで

## 注意メッセージ

### 使い方

```markdown
:::message
補足情報をここに書く
:::

:::message alert
重要な注意事項をここに書く
:::
```

### 使うタイミング

- **重要な補足**：message
- **注意喚起**：message alert
- **Tips**：message

### 例

```markdown
:::message
プライベートリポジトリにする場合は、GitHub Proが必要。
:::

:::message alert
**重要：スラッグは必ず指定しよう！**

指定しないとランダムな文字列になってしまう。
公開後は変更できないので要注意。
:::
```

## やってはいけないこと

### ❌ 避けるべき表現

- 堅苦しい敬語
- 専門用語の羅列（説明なし）
- 上から目線
- 「簡単です」「当然ですが」などの押し付け
- 完璧主義的な表現

### ❌ 避けるべき構成

- 前置きが長すぎる
- 結論が最後まで出てこない
- 見出しがない長文
- コードだけで説明がない

## 記事の長さ

### 目安

- **短め**：1000〜2000文字（Tips系）
- **中くらい**：2000〜3500文字（解説系）
- **長め**：3500文字〜（チュートリアル系）

### 長くなりすぎたら

- 前編・後編に分ける
- FAQを別記事にする
- 詳細は外部リンクに

- [ ] 失敗談や正直な感想が含まれている

- [ ] 絵文字は適度に使われている（多すぎない）
- [ ] 専門用語には説明がついている
- [ ] 読者目線で分かりやすいか
